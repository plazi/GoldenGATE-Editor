The Reasoners

Reasoners are Annotation sources that use existing Annotations to create new ones.

The Reasoners Menu

The Markup Converters menu provides access to functions for managing and applying Markup Converters, and access to a set of functions for direct markup conversion:
  • Environment Reasoning: Do reasoning over the Tokens neighboring Annotations of a particular type, comparing them to the neighboring Tokens of Annotations of another type.
  • Value Reasoning: Do reasoning over the values of Annotations of a particular type (the value of an Annotation is the text it encloses), finding new occurences of these values in the document.
  • Clone Annotations: Clone Annotations of a particular type, leaving aside other occurences of their values.
  • Value Environment Reasoning: Combination of value and environment reasoning, can be applied iteratively.
  • Token Fraction Reasoning: Do reasoning based on the individual Tokens enclosed in Annotations of a particular type, targeting Annotations of another type, regarding the fraction of Tokens enclosed by them that was also enclosed by Annotations of the first type.
  • Token Sequence Reasoning: Do reasoning based on the individual Tokens enclosed in Annotations of a particular type, finding new sequences of these Tokens.
  • Join Annotations: Join neighboring Annotations of the same type.
  • Apply: Apply some sort of reasoning to the document currently selected. This function is also available through the Apply Reasoner option in the Tools menu. Since there is only one Reasoner that acts in different ways, there is no selection list for Reasoners.

The Apply Reasoner Dialog

The dialog for Annotation reasoning pops up whenever reasoning is to be applied. It collects the necessary parameters for the particular reasoning mode selected, or additionally allows selecting the reasoning mode by choosing the coresponding parameter tab if no reasoning mode is selected initially.

Environment Reasoning

Environment reasoning makes use of Tokens neighboring the Annotations of a particular source type for qualifying Annotations of a particular target type, which is different from the source type. In particular, an Annotation of the target type is qualified if it has the same environment as some Annotation of the source type. Environment reasoning uses the following parameters:
  • Source Type: The type of the Annotations to collect the environments from.
  • Case Sensitive: Match the environments case sensitive or case insensitive.
  • Symmetric Environment: If checked, use the same number of Tokens to the left and right of the source type Annotations. Otherwise, use the Left Tokens and Right Tokens numbers.
  • Env(ironment) Tokens: For symmetric environments, the number of Tokens to collect to the left and right of the source type Annotations.
  • Left Tokens: For non-symmetric environments, the number of Tokens to collect to the left of the source type Annotations.
  • Right Tokens: For non-symmetric environments, the number of Tokens to collect to the right of the source type Annotations.
  • Target Type: The type of Annotations whose environments to compare to the environments of the source type Annotations.
Example: Assume using 2-Token symmertric environments of "location" Annotations for reasoning over "properName" Annotations. Further assume the following two sentences from some document (Annotations marked with XML tags):
  "When John F. Kennedy came to <location>Washinghton</location>, he was the first catholic to become president of the US."
  "When Adenauer first came to <properName>Bonn</properName>, he had just survived the war."
The 2-Token symmetric environment of "Washington" is "came to , he". Since this matches the 2-Token symmetric environment of "Bonn", the Reasoner would conclude that "Bonn" is a location, just as "Washington".

Value Reasoning

Value reasoning extracts the values of Annotations of a particular source type (the value of an Annotation is the text it encloses) and annotates all occurences of these values in the document. Value reasoning uses only two parameters:
  • Source Type: The type of the Annotations to collect the values from.
  • Case Sensitive: Match the values case sensitive or case insensitive.
Example: Assume value reasoning on "location" Annotations in the following two sentences from some document (Annotations marked with XML tags):
  "Every child knows that <location>Washinghton</location> is the capital of the USA."
  "When John F. Kennedy came to Washinghton, he was the first catholic to become president of the US."
Since "Washington" is annotated as a location in the first sentence, it would be annotated as a location in the second sentence as well.

Annotation Cloning

Annotation cloning simply duplicates the existing Annotations of a given type. Value reasoning uses only one parameter:
  • Source Type: The type of the Annotations to clone.
Case Sensitive is also displayed because it is necessary for all other reasoning modes, but has no influence.

Value / Environment Reasoning

Value / environment reasoning combines the two previous reasoning modes and applies them iteratively for a given number of rounds or until nither of them produces any new Annotations. It takes the same parameters as environment reasoning, plus the number of rounds:
  • Source Type: The type of the Annotations to collect the environments from.
  • Case Sensitive: Match the environments case sensitive or case insensitive.
  • Symmetric Environment: If checked, use the same number of Tokens to the left and right of the source type Annotations. Otherwise, use the Left Tokens and Right Tokens numbers.
  • Env(ironment) Tokens: For symmetric environments, the number of Tokens to collect to the left and right of the source type Annotations.
  • Left Tokens: For non-symmetric environments, the number of Tokens to collect to the left of the source type Annotations.
  • Right Tokens: For non-symmetric environments, the number of Tokens to collect to the right of the source type Annotations.
  • Rounds: Maximum number of iterations. Setting it to 0 will cause the reasoner to iterate until nither the value reasoning not the environment reasoning produces any new Annotations.
  • Target Type: The type of Annotations whose environments to compare to the environments of the source type Annotations.
Example: Assume using 2-Token symmertric environments of "location" Annotations for reasoning over "properName" Annotations. Further assume the following three sentences from some document (Annotations marked with XML tags):
  "Every child knows that <location>Washinghton</location> is the capital of the USA."
  "When John F. Kennedy came to Washinghton, he was the first catholic to become president of the US."
  "When Adenauer first came to <properName>Bonn</properName>, he had just survived the war."
Since "Washington" is annotated as a location in the first sentence, it will be annotated as a location in the second sentence as well. Then the 2-Token symmetric environments of "Washington" are "knows that is the" and "came to , he". Since the second one matches the 2-Token symmetric environment of "Bonn", the Reasoner would conclude that "Bonn" is also a location, just as "Washington". In the second round, the value reasoning would mmark every occurrence of "Bonn" as a location, leading to new environments for the next application of the environmental reasoning, leading to some new values again, and so on.

Token Fraction Reasoning

Token fraction reasoning uses the individual Tokens enclosed by Annotations of a particular source type to qualify the Annotations of of a particular target type, which is different from the source type. In particular, an Annotation of the target type is qualified if a given percentage of the Tokens enclosed in it were also enclosed in some Annotation of the source type. Token fraction reasoning needs four parameters:
  • Source Type: The type of the Annotations to collect the Tokens from.
  • Case Sensitive: Match the Tokens case sensitive or case insensitive.
  • Min % Known Tokens: Percentage of Tokens of a target type Annotation that have to be in some source type Annotation for the target type Annotation to qualify.
  • Target Type: The type of Annotations whose Tokens to compare to the Tokens collected from the source type Annotations.
Example: Assume applying Token fraction reasoning to the text passage below, using "person" Annotations as the source type, and targeting Annotations of type "properName" (Annotations marked with XML tags):
  "<person>Marilyn Monroe</person> once was the playmate of <person>John F. Kennedy</person>. I have read that at the bookstore of <properName>John Monroe<properName> around the corner."
The Tokens collected from the "person" Annotations would be {"F", "John", "Kennedy", "Merilyn", "Monroe"}. Since both "John" and "Monroe", the Tokens from the "properName" Annotation "John Monroe", are contained in this set, the Reasoner would conclude that "John Monroe" is a person, just as "Merilyn Monroe" and "John F. Kennedy".

Token Sequence Reasoning

Token sequence reasoning collects the Tokens enclosed in Annotations of a particular source type (referred to as known Tokens in the following) and annotates other combinations of these Tokens that occure elsewhere in the document. It can also span a small number of unknown Tokens (ones that were not enclosed in an Annotation of the source type) if they stand between to known ones, and it can use a gazetteer list to filter these unknown Tokens. The parameters for Token sequence reasoning are:
  • Source Type: The type of the Annotations to collect the known Tokens from.
  • Case Sensitive: Match the Tokens case sensitive or case insensitive.
  • Max Tokens to Span: Maximum number of unknown Tokens to span if standing between two known ones.
  • Token List: Gazetteer list (see Lists) for filtering the unknown Tokens (relevant only if prevoius parameter > 0).
  • Use List for Inclusion: If checked, unknown Tokens are spanned only if they are contained in the gazetteer list. Otherwise, unknown Tokens are spanned only if they are not contained in the gazetteer list.
Example: Assume applying Token sequence reasoning over the text passage below, using "person" Annotations as the source type, spanning a maximum of 2 Tokens (Annotations marked with XML tags):
  "<person>Jacky Kennedy Onassis</person> was the wife of John F. Kennedy. I have read that at the bookstore of <person>John Monroe<person> around the corner."
The Tokens collected from the "person" Annotations would be {"Jacky", "John", "Kennedy", "Monroe", "Onassis"}. Of "John F. Kennedy", "John" and "Kennedy" are known Tokens now, and since we span 2 Tokens at maximum, "F." will bespanned, resulting in "John F. Kennedy" being annotated as a person.

Annotation Joining

Annotation joining works pretty similar to Token sequence reasoning, except for the fact that it does not span unknown Tokens between known Tokens, but only between Annotations of the source type. It also can use a gazetteer list to filter the unknown Tokens. The parameters for Token sequence reasoning are:
  • Source Type: The type of the Annotations to join.
  • Case Sensitive: Do the gazetteer list lookups case sensitive or case insensitive.
  • Max Tokens to Span: Maximum number of unknown Tokens to span if standing between two Annotations of the source type.
  • Token List: Gazetteer list (see Lists) for filtering the unknown Tokens (relevant only if prevoius parameter > 0).
  • Use List for Inclusion: If checked, unknown Tokens are spanned only if they are contained in the gazetteer list. Otherwise, unknown Tokens are spanned only if they are not contained in the gazetteer list.
Example: Assume joining Annotations of type "location" in the Text passage below, spanning a maximum of 2 Token (Annotations marked with XML tags):
  "<location>Port of Spain</location> is the capital of <location>Trinidad</location> and <location>Tobago</location>."
The Annotations marking "Trinidad" and "Tobago" would be joined, spanning the intermediate "and", and resulting in a single Annotation enclosing "Trinidad and Tobago". The Annotations marking "Port of Spain" and "Trinidad", however, would not be joined because between them, there are the four Tokens "is", "the", "capital", and "of", and we are spanning a maximum of 2 Tokens.
